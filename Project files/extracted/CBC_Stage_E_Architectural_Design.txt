1. Introduction
1.1 Purpose of Stage E
Stage E of the Unified Software Architectural Process — Architectural Design — builds directly on the
behavioural models produced in Stage D. Where Stage D defined how objects interact and transition
over time, Stage E defines where those objects and interactions live: the structural blueprint of the
system, how components are assembled, how they communicate, and how they are deployed in a real-
world cloud environment.
This document — the Software Architecture Document (SAD) — is the authoritative output of Stage E.
It establishes the architectural decisions that will govern all subsequent stages, including database
design (Stage F) and UI/UX design (Stage G).
1.2 Architectural Goals
The CBC Digital Learning Platform architecture is designed to satisfy the following goals derived
directly from the SRS v1.0:
• Serve 100,000 concurrent learners nationally without degradation (NFR-S-01, NFR-S-02)
• Operate reliably on 2G/3G connectivity in rural Uganda (NFR-P-01, NFR-P-03)
• Support offline lesson access through a progressive web application strategy (NFR-O-01 to
NFR-O-03)
• Ground AI tutor responses in NCDC domestic curriculum content via MCP (FR-AI-02, FR-AI-03)
• Enforce strict role-based access control and learner data privacy by default (NFR-SEC-04,
NFR-SEC-05)
• Maintain 99% uptime during school hours with graceful degradation of the AI layer (NFR-A-01,
NFR-A-03)
1.3 Architectural Approach: Layered Monolith
The platform adopts a Layered Monolith architecture. This decision is deliberate and appropriate for the
following reasons:
• Prototype and academic scope: a monolith delivers the full system without the operational
overhead of service orchestration, container registries, and inter-service networking that
microservices demand.
• Single deployment unit: the entire backend deploys as one Django application, simplifying
CI/CD pipelines, debugging, and academic assessment.
• Clear layer separation: the monolith is internally structured into four strict layers — Presentation,
Application, AI, and Data — which enforces clean separation of concerns while keeping
integration simple.
• Upgrade path: the layered structure allows the AI layer and individual application services to be
extracted into standalone microservices in future phases without rewriting business logic.
The four layers are: Presentation (React SPA), Application (Django/DRF backend services), AI (AI
Tutor + MCP Server), and Data (PostgreSQL, Redis, S3, IndexedDB).

2.2 Layer Descriptions
Layer 1 — Presentation Layer
The Presentation Layer is a React Single-Page Application (SPA) delivered to the browser. It
comprises four functional modules: the LearnerApp (content browsing, AI tutor interface, posts and
collaboration), the TeacherApp (material upload and management), the MoES/NCDC Admin
Dashboard (curriculum management and analytics), and the OfflineModule (a Progressive Web
Application service worker paired with browser-side IndexedDB for offline lesson access).
All modules communicate with the backend over HTTPS REST (port 443) via Axios, and over
WebSocket (Django Channels) for real-time collaboration. The SPA is built as a static production build
and served via AWS CloudFront CDN with origin in S3, ensuring low-latency delivery across Africa.
Layer 2 — Application Layer
The Application Layer is the Django/DRF backend. It is organised into eight logically separated Django
applications, each owning a distinct domain of business logic. These are: AuthService (registration,
login, JWT issuance, RBAC), ContentService (CBC lessons and teacher materials with
subject/level/competency tagging), PostService (learner posts, comments, reactions), CollabService
(real-time collaborative sessions via Django Channels WebSocket), ProgressService (competency
attainment tracking against the CBC framework), AnalyticsService (national aggregated reporting for

MoES/NCDC), OfflineSyncService (packaging and delta-syncing lesson content for offline download),
and MediaService (file upload and AWS S3 presigned URL management).
The backend runs on Gunicorn (HTTP) and Daphne (ASGI for WebSocket), behind an Nginx reverse
proxy for SSL termination. Celery with Redis as broker handles asynchronous tasks such as analytics
aggregation and notification dispatch.
Layer 3 — AI Layer
The AI Layer sits between the Application Layer and the external AI inference service. It consists of two
components: the AITutorService and the MCP Server. The AITutorService receives a learner query
from the Application Layer, retrieves relevant structured content from the NCDC knowledge base via
the MCP Server, builds a context-enriched prompt, and dispatches it to the External AI API. The
response is returned to the learner and logged as an AISession record in PostgreSQL.
The MCP (Model Context Protocol) Server acts as a structured query interface over the NCDC
curriculum content stored in PostgreSQL. It allows the AI model to access specific CBC content —
subject material, competency definitions, and learning objectives — in a controlled, structured manner,
ensuring responses are grounded in Ugandan domestic academic content rather than generic foreign
knowledge bases.
The external AI inference provider is Claude (Anthropic) or OpenAI, accessed over HTTPS with API
key authentication. This call is outbound only and does not expose any internal platform data beyond
the constructed prompt.
Layer 4 — Data Layer
The Data Layer consists of four storage components. PostgreSQL (AWS RDS, af-south-1) is the
primary relational database storing all domain objects defined in Stage C: users, learner profiles,
lessons, posts, AI sessions, competency progress records, and the NCDC knowledge base. Redis
(AWS ElastiCache) handles session tokens, JWT blacklisting, Celery task brokering, and Django
Channels pub/sub for WebSocket message routing. AWS S3 stores all binary media — uploaded
teacher materials, images, and packaged offline content bundles. IndexedDB is the browser-side offline
store, managed by the PWA service worker to hold pre-downloaded lesson content for connectivity-free
access.
Cross-Cutting Concerns
Several concerns apply uniformly across all layers: all traffic uses HTTPS/TLS 1.2 or higher; JWT
tokens enforce role-based access control at every API endpoint; Django Channels provides the
WebSocket infrastructure for real-time features; Celery processes all async operations; and
CloudWatch provides centralised logging and alerting.

4. Deployment Diagram
4.1 Deployment Environment
The platform is deployed on Amazon Web Services in the af-south-1 region (Cape Town, South Africa).
This region was selected as the closest AWS region to Uganda, minimising network latency for East
African users compared to using European or US-based regions. All compute, database, and cache
resources reside within a Virtual Private Cloud (VPC) with private subnets, ensuring no direct public
internet access to backend infrastructure.

5. API Contracts
5.1 API Design Principles
The platform exposes a RESTful JSON API served by Django REST Framework on all routes under
/api/v1/. Authentication uses JWT Bearer tokens issued by the AuthService. All endpoints enforce
RBAC: each route declares which roles (learner, teacher, moES_official) are permitted. WebSocket
connections for real-time collaboration and notifications are served by Django Channels under the /ws/
path prefix.
API versioning is path-based (/api/v1/) to allow future API evolution without breaking existing clients. All
responses follow a consistent envelope: { "status": "success"|"error", "data": {...}, "message": "..." }.
