1. Introduction
1.1 Purpose of Stage F
Stage F of the Unified Software Architectural Process — Database & Data Design — derives the full
relational database schema from the domain model produced in Stage C. Where Stage C identified the
domain entities and their conceptual relationships, Stage F translates those entities into concrete
database tables, columns, data types, constraints, indices, and foreign key relationships suitable for
implementation in PostgreSQL on AWS RDS.
This document constitutes the complete Data Design deliverable for the CBC Digital Learning Platform.
It includes the Entity Relationship Diagram (ERD) in crow's foot notation, the full relational schema with
a data dictionary for every table, and the data storage and offline synchronisation strategy.
1.2 Inheritance Pattern Decision
The User hierarchy (User → Learner, Teacher, MoESOfficial) and Content hierarchy (Content →
Lesson, TeacherMaterial) are implemented using Joined Table Inheritance. This means a base table
(users, content) holds all shared columns, and each subclass has its own extension table with a
primary key that is simultaneously a foreign key back to the base table. This pattern:
• Eliminates NULL columns that would appear in Single Table Inheritance
• Aligns exactly with Django's multi-table inheritance ORM, requiring no custom mapping
• Keeps each table's responsibility clear and query-able independently
• Maintains referential integrity at the database level
1.3 Design Conventions
• All primary keys use UUID (gen_random_uuid()) — avoids integer sequence exposure in URLs
and supports future distributed sharding
• All timestamps use TIMESTAMPTZ (timestamp with time zone) stored in UTC — correct for a
Uganda-deployed platform accessed from multiple time zones
• Soft deletes: sensitive tables (users, posts) use is_active / is_deleted flags rather than physical
deletion, preserving audit trails
• All ENUM types are defined as PostgreSQL custom ENUM types for type safety
• Junction tables use composite primary keys (both FKs combined) to enforce uniqueness at the
DB level
• Indices are defined on all foreign keys and all frequently-filtered columns per the query patterns
in the SRS

4. Data Flow & Offline Sync Strategy
4.1 Overview Diagram
Figure F-02 presents the complete data flow for both online and offline paths, the five-step offline sync
process, the storage allocation strategy by data type, and the feature availability matrix per connectivity
state.
FigureF-02:CBCDigitalLearningPlatform—DataFlow&OfflineSyncStrategy
4.2 Online Data Flow
In normal connected operation, the learner's React SPA communicates with the Django API over
HTTPS via CloudFront and the Application Load Balancer. The Django backend reads and writes to
PostgreSQL via the ORM for all persistent domain data, reads from and writes to Redis for session
tokens, Celery task brokering, and WebSocket channel state, and reads from and writes to S3 for
media files and generated reports. All static assets and the SPA build are served from S3 via
CloudFront edge nodes, minimising latency for learners in Uganda relative to the AWS af-south-1
origin.
4.3 Offline Sync Flow
The offline sync process is a five-step sequence initiated by the learner while connected:
• Step 1 — Request: The learner taps "Download Lesson" in the LearnerApp UI while online. The
request hits /api/v1/content/offline-bundle/ with the lesson ID.
• Step 2 — Package: The OfflineSyncService in Django retrieves the lesson body, associated
assets, and metadata from PostgreSQL and S3, and packages them into a self-contained HTML
bundle.
• Step 3 — Store: The bundle is saved to the S3 offline bundles prefix. An offline_cache record is
written to PostgreSQL with sync_status = SYNCED. An offline_sync_log entry is created.
• Step 4 — Deliver: The bundle URL is returned to the React SPA. The PWA Service Worker
intercepts the response and stores the bundle in browser IndexedDB.
